How everybody should use git

Architecture

Every project should have these branches:

  • master, and this branch has to stay the “clean” branch, corresponding to what’s running in production
  • dev, and this branch also has to stay “clean” because it’s the branch everybody is going to use in order to implement a new functionality.

From this point, every functionality has to be developed in a separate branch, coming from the dev branch, and therefore has to be merged with the dev branch. About the naming, the branch should be called in order to be able to express herself what content we may find on it (for example, a branch to review all the UX should be called review-ux)

From the above diagram, we see that every change is operated on a different branch, to be then merged onto the dev branch, which happens to one of the only one to be merged into the master branch.

In this case, the team realized after propagating a new release that they had a typo error. They decided to create a branch, from the master, operate the change, and propagate this change to the master and the dev branch.

Commits

Every change has to be committed. Indeed, the more specific your commits are, the easier it is to eventually review the code in case of bugs.

Commits should be named correctly, and should precise as best as possible what they are about, and if needed, refer to the issue concerning the functionality or the bug being treated.

Let’s say we are working around an issue referenced with id 42. To make a reference to an issue in a commit, we simply need to write “#42” in the commit message, and the UI makes the reference itself.

Issues

Issues should be used as much as possible. It keeps track of every bug, and a simple search allows the team to go through the changes that have been operated to fix any bug that has ever been encountered.

To reference a commit into an issue, the best thing to do is to copy-paste the SHA1 associated to the commit.

If you want to refer an issue to another, writing “#42” to refer the issue with the id 42 is the way to do.

Pull requests

Every merge should be operated through the use of pull requests. It allows the entire team to talk about it, and it keeps track on the UI about it.
